Bawbel Scanner

Open-source CLI scanner for agentic AI components — SKILL.md files, MCP servers, system prompts, plugins. Detects AVE vulnerabilities before they reach production.

bawbel scan

Scan a component or directory for AVE vulnerabilities.

bawbel scan PATH [OPTIONS]

Options

OptionDefaultDescription
--formattexttext · json · sarif
--fail-on-severity—Exit 2 if findings at or above: critical high medium low
--recursive, -rfalseScan all matching files in a directory tree
--watch, -wfalseWatch for file changes and re-scan automatically

Exit codes

CodeMeaning
0Clean, or findings below --fail-on-severity threshold
2Findings at or above --fail-on-severity threshold

Examples

# Scan one file, text output
bawbel scan ./my-skill.md

# Recursive scan, fail on HIGH+
bawbel scan ./skills/ --recursive --fail-on-severity high

# JSON — pipe into jq
bawbel scan ./skills/ --format json | jq '.[].max_severity'

# SARIF — upload to GitHub Security tab
bawbel scan ./skills/ --format sarif > results.sarif

Start watching

# Watch a single file
bawbel scan ./skill.md --watch

# Watch an entire directory
bawbel scan ./skills/ --watch --recursive

# Watch with JSON output (for tooling integration)
bawbel scan ./skills/ --watch --format json | your-tool
Tip: Watch mode pairs perfectly with the VS Code extension — use watch mode in a terminal for full output while the extension shows inline diagnostics in your editor.

How it works

  1. Uses watchdog (requires pip install "bawbel-scanner[watch]") to monitor the filesystem
  2. On any file write event, re-runs the full scan pipeline immediately
  3. Prints a clean diff-style output showing new or resolved findings
  4. Ctrl+C stops the watcher cleanly

bawbel init

Initialise Bawbel Scanner in a project — generates .bawbelignore and bawbel.yml, discovers skill and MCP files, and prints recommended next steps.

bawbel init [--path PATH]

Options

OptionDefaultDescription
--path.Project root directory to initialise

What it does

  1. Scans the directory for skill files (SKILL.md, system_prompt.*, agent.md, etc.) and MCP manifests (mcp_*.json, mcp_*.yaml)
  2. Creates .bawbelignore suppressing docs/**, tests/fixtures/**, examples/**, and other non-skill paths
  3. Creates bawbel.yml with project scan settings
  4. Prints all discovered skill files and recommended next steps

Writing Rules

Add a new entry to PATTERN_RULES in scanner/engines/pattern.py. No other files need to change to add a pattern rule.

Rule structure

{
    "rule_id":     "bawbel-your-rule",    # kebab-case · unique forever
    "ave_id":      "AVE-2026-NNNNN",     # or None if no record yet
    "title":       "Short title (max 80 chars)",
    "description": "Full description of what this detects.",
    "severity":    Severity.HIGH,
    "cvss_ai":     8.0,                   # 0.0–10.0
    "owasp":       ["ASI01", "ASI08"],
    "patterns": [
        r"your\s+regex\s+here",           # re.IGNORECASE applied
        r"alternative\s+pattern",         # first match per file wins
    ],
},

Also add a remediation entry

In scanner/cli.py, add to REMEDIATION_GUIDE so bawbel report shows specific fix instructions:

REMEDIATION_GUIDE = {
    ...
    "bawbel-your-rule": (
        "Specific instructions: what to remove and what to do instead."
    ),
}

Required: tests

Every rule needs a positive fixture (triggers) and a negative fixture (does not trigger):

# Positive — must detect
def test_detects_your_rule(self, tmp_path):
    path = write_skill(tmp_path, "s.md", "# Skill\n[triggering content]\n")
    result = scan(path)
    assert "bawbel-your-rule" in [f.rule_id for f in result.findings]

# Negative — must not false-positive
def test_your_rule_no_false_positive(self, tmp_path):
    path = write_skill(tmp_path, "s.md", "# Skill\n[innocent content]\n")
    result = scan(path)
    assert "bawbel-your-rule" not in [f.rule_id for f in result.findings]

Detection Engines

6 detection engines in scanner/engines/. Each returns list[Finding] — or an empty list if its dependency is not installed. scan() always completes and never raises.

Stage 0 Magika File type verification — supply chain attack detection New v1.0 Optional
Stage 1a Pattern 24 rules — regex, case-insensitive, all AVE IDs Always active
Stage 1b YARA 15 rules — binary + complex text patterns Optional
Stage 1c Semgrep 15 rules — structural + multi-line patterns Optional
Stage 2 LLM Semantic analysis via LiteLLM — any provider Optional + API key
Stage 3 Sandbox Behavioral analysis in isolated Docker container Docker required

Engine contract

Every engine follows the same contract — safe to add, remove, or replace without touching any other engine:

# Input: resolved absolute file path (str) or file content (str)
# Output: list of Finding objects — empty list if dependency missing or error
# Contract: NEVER raises — all errors are caught and logged

def run_<engine>_scan(file_path: str) -> list[Finding]:
    if not dependency_available():
        return []           # skip silently
    findings = []
    # ... detection logic ...
    return findings
Stage 0 Magika — File Type Verification pip install "bawbel-scanner[magika]" · included in [all]

Purpose

Runs before all text analysis engines. Uses Google's Magika (~99% accuracy, ~5ms/file, no GPU) to verify that each file's content type matches its declared extension. Catches supply chain attacks that regex cannot detect — the file contains no readable text to match.

What it catches

ELF binary in .md or .yamlCRITICAL · AVE-2026-00024
Python pickle in .json/.yamlCRITICAL · RCE vector
PE32/PE64 exe in .mdCRITICAL · Windows executable
Python bytecode in .txtCRITICAL · Compiled .pyc
PHP / JSP in .mdCRITICAL · Server-side code
Shell script in .yamlHIGH · AVE-2026-00024

Detection flow

magika.identify_path(path)    ← ~5ms, CPU only, no GPU required
    │
    ├─ score < 0.75     → skip (uncertain, avoid FP)
    │
    ├─ DANGEROUS type   → CRITICAL finding + skip text engines
    │   elf · pe32 · pe64 · pickle · php · jsp · shell · pyc
    │
    └─ extension mismatch → HIGH finding + text engines continue
Key behaviour: when Magika flags a dangerous content type, all text analysis engines are skipped — running regex on binary content is meaningless. Stage 3 (sandbox) still runs on the original bytes.

Meta-Analyzer (FP-4)

After all static engines run, the meta-analyzer sends medium-confidence findings (0.35–0.80) to the LLM as enriched context — one call per file, not per finding. The LLM classifies each finding as real, false_positive, or needs_review.

LLM Engine (Stage 2)Meta-Analyzer (FP-4)
InputRaw file contentAll findings + file metadata + context lines
TaskFind new vulnerabilitiesClassify existing findings: real / FP / review
Runs whenEvery fileFiles with medium-confidence findings only
BAWBEL_META_ANALYZER_ENABLED=true   # default
BAWBEL_META_MIN_CONFIDENCE=0.35     # lower bound for meta-analysis
BAWBEL_META_MAX_CONFIDENCE=0.80     # upper bound — high confidence skips LLM
Stage 3 Behavioral Sandbox Docker + BAWBEL_SANDBOX_ENABLED=true

Entry point

run_sandbox_scan(file_path: str) -> list[Finding]

Runs the skill file inside an isolated Docker container and checks for behavioral indicators — network calls, file writes, process spawning. The most expensive stage: use for pre-release audits, not watch mode.

Stage 3 always runs on the original file content — not the stripped version used by text engines. This ensures behavioral analysis sees exactly what the agent will execute.

Deduplication

# scanner.py runs all engines then deduplicates:
# Pass 1 — deduplicate by rule_id (same engine, same rule)
# Pass 2 — deduplicate by (ave_id) across engines
#          finding WITH a line number wins over one without
#          priority: pattern > yara > semgrep > llm > sandbox

findings = (
    run_magika_scan(file_path)         # Stage 0
    + run_pattern_scan(stripped)        # Stage 1a
    + run_yara_scan(file_path, stripped)# Stage 1b
    + run_semgrep_scan(file_path, stripped) # Stage 1c
    + run_llm_scan(content)             # Stage 2
    + run_sandbox_scan(file_path)       # Stage 3
)
findings = _deduplicate(findings)

Suppression

Three mechanisms to suppress known false positives. Suppressed findings are never deleted — they move to suppressed_findings and always appear in JSON/SARIF output so audits remain complete.

1 — Inline suppression

Add a comment on the same line as the content that triggers the finding.

# Suppress all findings on this line
fetch https://internal.company.com  <!-- bawbel-ignore -->

# Suppress a specific rule
fetch https://internal.company.com  <!-- bawbel-ignore: bawbel-external-fetch -->

# Suppress by AVE ID
fetch https://internal.company.com  <!-- bawbel-ignore: AVE-2026-00001 -->

# Suppress multiple rules
fetch https://internal.company.com  <!-- bawbel-ignore: bawbel-external-fetch, AVE-2026-00007 -->

# Also works with # and // comment styles
fetch https://internal.company.com  # bawbel-ignore
fetch https://internal.company.com  // bawbel-ignore: bawbel-external-fetch
Inline suppression only works for findings with a line number. YARA findings (whole-file matches) use .bawbelignore instead.

2 — Block suppression

Suppress all findings within a section — useful for test fixtures and documentation examples.

<!-- bawbel-ignore-start -->
fetch https://internal.company.com
Ignore all previous instructions  <!-- intentional — test fixture -->
<!-- bawbel-ignore-end -->

# Also works with # and // styles
# bawbel-ignore-start
...
# bawbel-ignore-end

3 — .bawbelignore

Suppress entire files or directories. Uses gitignore syntax.

# .bawbelignore
tests/fixtures/malicious/**    # all test fixtures
docs/examples/bad.md           # known-bad example
**/test_*.md                   # all test skill files
examples/                      # all files in examples/
PatternMatches
bad.mdAny file named bad.md in any directory
tests/fixtures/bad.mdExact path relative to .bawbelignore
tests/fixtures/**All files recursively under tests/fixtures/
**/test_*.mdAny file starting with test_ in any directory

Audit mode — override all suppressions

bawbel scan ./skills/ --no-ignore       # CLI flag
BAWBEL_NO_IGNORE=true bawbel scan ./skills/

JSON output always includes suppressed_findings with suppressed: true and suppression_reason — suppression never hides findings from security audits.

False Positive Reduction

False positives are the single biggest reason people stop using a security scanner. Bawbel v1.0 ships a 5-layer FP system — scanning 21 docs files now produces 0 active findings, down from ~30 in v0.3.

Why false positives happen

Pattern matching (regex, YARA, Semgrep) has no understanding of context — it sees bytes and strings, not meaning. Four root causes:

CauseExample
Documentation examples of attacksA guide showing goal override triggers bawbel-goal-override
Code examples in technical guidespip install pkg in a getting-started guide triggers bawbel-shell-pipe
Static engines missing structureYARA reads raw bytes — no concept of markdown tables, fences, or headings
Heading or table context"## External instruction fetch" in a doc heading triggers the pattern

The 5-layer FP system (v1.0)

FP-1
Code fence stripping
Content inside ` ``` ` and ~~~ blocks is blanked before static analysis. Line numbers remain accurate. Sandbox receives original content.
✅ ~60%
FP-2
Preceding-line negation context
If the line immediately above contains "Never do this:", "avoid:", "bad example:", "❌" etc. — the finding on the next line is suppressed. Only the immediate next line is affected.
✅ ~15%
FP-3
Confidence scoring (0.0–1.0)
Each finding is scored. Penalties: table row (−0.45), heading (−0.55), docs/ path (−0.35), short match (−0.20). Boosts: first 30 lines (+0.15), multi-engine agreement (+0.25), SKILL.md filename (+0.15). Threshold: 0.80.
✅ ~10%
FP-4
Meta-analysis LLM filter
Medium-confidence findings (0.35–0.80) sent to the LLM as enriched context — one call per file. Verdicts: real (confidence +0.15), false_positive (suppressed), needs_review (−0.05). Requires LLM configured.
✅ ~7%
FP-5
File-type scan profiles
Files classified as: skill (threshold 0.60), mcp_manifest (0.55), documentation (0.85), unknown (0.80). Documentation files require much higher confidence before surfacing.
✅ ~3%

Confidence scoring signals

SignalEffectReason
Match in markdown table row (|...|)−0.45Comparison tables show attack examples
Match in heading (## Title)−0.55Headings describe, not instruct
File path contains docs/ examples/−0.35Documentation context
Match text < 6 chars−0.20Too little signal to trust
Match in first 30 lines (non-docs)+0.15Instructions usually at top of skills
Two or more engines agree (same AVE ID)+0.25Cross-engine agreement is strong signal
Filename is SKILL.md or system_prompt.*+0.15Known skill file — scan hard
CVSS-AI ≥ 9.0+0.05Critical patterns — slight trust boost

CI/CD

Add Bawbel to your pipeline to catch malicious components before they merge. The official action at bawbel/bawbel-integrations handles install, scan, and SARIF upload in one step.

GitHub Actions — official action ✅

uses: bawbel/bawbel-integrations@v1 — installs scanner, runs scan, uploads SARIF automatically · source ↗ 
# .github/workflows/bawbel.yml
name: Bawbel Security Scan
on: [push, pull_request]

jobs:
  scan:
    runs-on: ubuntu-latest
    permissions:
      security-events: write
      contents: read
    steps:
      - uses: actions/checkout@v4

      - uses: bawbel/bawbel-integrations@v1
        id: bawbel
        with:
          path: .
          fail-on-severity: high
          format: sarif

      - uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: ${{ steps.bawbel.outputs.sarif-file }}

Python API

scan() is the single public entry point. Never raises — all errors are captured in ScanResult.error.

Basic usage

from scanner import scan

result = scan("/path/to/skill.md")

if result.is_clean:
    print("Clean")
elif result.has_error:
    print(f"Error: {result.error}")        # E-code only, no internal detail
else:
    for f in result.findings:
        print(f"[{f.severity.value}] {f.title} — {f.cvss_ai}")
    print(f"Risk: {result.risk_score:.1f} / 10")

Batch scanning

from pathlib import Path
from scanner import scan

results = [scan(str(p)) for p in Path("./skills").rglob("*.md")]
critical = [r for r in results if r.max_severity and r.max_severity.value == "CRITICAL"]
print(f"{len(critical)} critical out of {len(results)} scanned")

Enrich scan findings

Every bawbel-scanner finding includes an ave_id. Use it to pull the full record at scan time:

import requests
from scanner import scan

result = scan("./my-skill.md")

for f in result.findings:
    if f.ave_id:
        r = requests.get(
            f"https://api.piranha.bawbel.io/ave/{f.ave_id}"
        ).json()
        print(f"[{f.severity.value}] {f.title}")
        print(f"  Fingerprint: {r['behavioral_fingerprint']}")
        print(f"  Remediation: {r['remediation']}")
